## 1. システム概要

**目的**: 建設派遣業「マツケン」の業務を支える配置管理 SaaS。既存ツール「サポスケ」の代替として開発。

**主要ユースケース**:

| 業務 | 主担当ロール | 該当画面 |
|---|---|---|
| 配置作成（誰をいつどこへ） | admin / office | `/calendar` |
| 配置の競合・保険要件チェック | admin / office | 配置時の警告 |
| スタッフ／現場／得意先／車両のマスタ管理 | admin | `/staff` `/sites` `/customers` `/vehicles` |
| 日次の支払・相殺計算 | admin | `/tally` |
| 出来高確認書（施主サイン入り） | admin / office / staff | `/forms` |
| 作業日報の印刷 | admin / manager / office | `/print/work-report` |
| スタッフ本人による予定確認 | staff | `/calendar`（限定ビュー） |
| QR コードでの簡易ログイン | 全員 | `/login/qr` |
| 監査ログ・変更履歴 | admin | `/audit-logs` |
| 営業所マスタ管理 | admin | `/settings` |

**フェーズの位置付け**: 2026/04 デモ → 5/20 本番デモ → 漸進的に拡張中。`SELECTABLE_USER_ROLES` を `admin / office / staff` の 3 つに絞っているのは 5/20 デモ向けの簡素化（`constants.ts:69-80`）。

**現状の営業所**: 守口 / 高瀬 / 守口第二 / 橋波（直近 2026/05/19 の作業で seed の本社・草津・京都を撤去し、新体制へ差し替え済み）。

## 4. 環境構成 — dev / Turso 本番

### 4.1 二刀流の Prisma クライアント（`src/lib/prisma.ts:53-66`）

```ts
function buildPrisma(): PrismaClient {
  const tursoUrl = process.env.TURSO_DATABASE_URL;
  if (tursoUrl) {
    const adapter = new PrismaLibSQL({
      url: tursoUrl,
      authToken: process.env.TURSO_AUTH_TOKEN,
    });
    return new PrismaClient({ adapter });
  }
  return new PrismaClient();
}
```

- `TURSO_DATABASE_URL` が **環境変数に存在すれば** Turso（libSQL HTTP）に接続。なければ `DATABASE_URL`（ローカル `file:./dev.db`）。
- Next.js 開発時は `.env` を読むので **dev でも Turso 本番に接続される**点に注意（記憶: `[DB は Turso 共有・migrate はローカル dev.db のみ更新]`）。

### 4.2 マイグレーションの取り扱い

- `prisma migrate dev` で生成した SQL は **ローカル dev.db にしか適用されない**
- **本番 Turso へは `turso db shell matsken-prod` で手動 ALTER** が必要（運用メモ参照）
- そのため `prisma/migrations/` ディレクトリは「ローカル仕様の履歴」と「本番への手順書」を兼ねる扱い

### 4.3 環境変数（`.env`）

| キー | 役割 |
|---|---|
| `DATABASE_URL` | ローカル SQLite (`file:./dev.db`) |
| `SESSION_SECRET` | iron-session 暗号鍵（最低 32 文字） |
| `TURSO_DATABASE_URL` | 本番 libSQL URL |
| `TURSO_AUTH_TOKEN` | Turso 認証 JWT |

### 4.4 ローカル起動

```
npm run dev          # next dev（Turbopack）
npm run db:seed      # tsx prisma/seed.ts でデモデータ投入
npm run db:reset     # migrate reset --force（全消し→再 migrate）
npm run build        # prisma generate && next build
```

`postinstall: prisma generate` のため `npm i` の段階で Prisma クライアントは自動生成される。

## 5. 認証・セッション・ロール体系

### 5.1 セッション仕様（`src/lib/session.ts`）

| 項目 | 値 |
|---|---|
| Cookie 名 | `matsken_session` |
| TTL | 1 週間（`60 * 60 * 24 * 7`） |
| httpOnly | true |
| secure | production のみ |
| sameSite | lax |
| Path | `/` |
| ストア | iron-session（暗号化された Cookie 本体に状態を保存、外部ストア不要） |

`SessionData = { userId: number }` のみ。役割や branch は毎リクエスト DB 引き直し。

### 5.2 ログイン経路（`src/lib/auth.ts`）

3 つのエントリポイント:

1. **`login(username, password)`** — `users.passwordHash` を bcrypt で比較。
2. **`loginByToken(token)`** — `users.loginToken` に紐付くユーザーで認証（QR）。トークンは `users/[id]/regenerate-qr` で `base64url(24bytes)` を発行・差し替え。
3. **`getSession()`** — Cookie から `userId` を取り出し DB lookup。`isActive=false` なら null。

3 経路すべてで **`setAuditActor()`** を呼び、後段の Prisma middleware が監査ログを記録できるようにしている。

### 5.3 ロール体系（`constants.ts:69-81`）

DB 上は 5 ロール:
- `admin` — 管理者（全権、削除可、お金関連 OK）
- `manager` — 所長（manager API role を要求する箇所では admin と同等扱いの場合あり）
- `office` — ユーザー 1。日計表など金額編集は不可だが、配置・現場・スタッフは編集可
- `viewer` — 閲覧専用
- `staff` — 本人。自分のスタッフ ID に紐付くデータのみ可視

UI のプルダウンでは `SELECTABLE_USER_ROLES = ["admin", "office", "staff"]` のみが選択肢（5/20 デモ向け簡素化）。`manager` / `viewer` は DB 上は許容するが選ぶ手段が無い。

### 5.4 ロールごとの主な制約

| ロール | 配置作成/編集 | 日計表編集 | スタッフ／現場マスタ削除 | 監査ログ | 設定 |
|---|---|---|---|---|---|
| admin | ✅ | ✅ | ✅ | ✅ | ✅ |
| manager | ✅ | ❌ (閲覧のみ) | ❌ | ❌ | ❌ |
| office | ✅ | ❌ (閲覧のみ) | ❌ | ❌ | ❌ |
| viewer | ❌ | ❌ | ❌ | ❌ | ❌ |
| staff | 自分のみ確認/「見ました」 | ❌ | ❌ | ❌ | ❌ |

`api-auth.ts` の `requireAuth()` / `requireRole(...roles)` で各 API が判定。ページ側は `getSession()` 直後に `if (session.role !== "admin") redirect("/calendar")` パターン。

### 6.4 Staff（スタッフ）

40 弱のフィールド。主要なもの:

```
employeeCode      Unique     "M001" 等
branchOfficeId    FK
name / nameKana / displayName
phone
insuranceType     [DEPRECATED] "company"/"national"/"sole_proprietor"
hasShaho          Bool       社保加入   ← 新スキーマ
hasKokuho         Bool       国保加入
hasIchiriOyakata  Bool       一人親方
residenceType     "dorm1"=旧寮 / "dorm2"=新寮 / "commuter"=通い
role              "admin"/"manager"/"office"/"worker"  ← User.role とは別
dailyRate         Int?       基本日当
isActive          Bool       論理削除フラグ
licenseExpiry     String?    免許期限 (YYYY-MM-DD)
openingBalance    Int        期首残高（日計表の累計起点）
openingBalanceDate String?   この日以降を集計対象とする
notes
```

**重要**:
- 削除は **論理削除**（`isActive=false`）。物理削除はしない（過去配置の参照を壊さないため）。
- `insuranceType` は互換のため残置。**新しいコードは `hasShaho`/`hasKokuho`/`hasIchiriOyakata` の 3 Bool で判定**（schema.prisma:53）。
- `openingBalance` / `openingBalanceDate` の組で「導入時の前期繰越を尊重しつつ、その日以降を累計」するルール（後述）。

### 6.5 Customer（得意先 親） & JobSite（現場 子）

議事録 L131: 「得意先を管理するマスター…そいつの小要素に現場を持たせる」

- `Customer`: code（任意 unique）, name, address, phone, notes, isActive
- `JobSite`: `customerId` で Customer に連結
- **移行期のため `JobSite.clientCode` / `clientName` も残置**（Customer マスタ化前の自由入力分を保持）

`JobSite` の特徴的なフィールド:

| フィールド | 意味 |
|---|---|
| `siteCode` | Unique |
| `branchOfficeId` | 担当営業所 |
| `mapUrl` | Google Map URL（印刷時 QR 化） |
| `contactName{1,2,3}` / `contactTel{1,2,3}` | 連絡先 3 枠 |
| `transportation` | 交通手段（自由記述、配置作成時に prefill） |
| `belongings` | 持ち物（同上） |
| `siteMemo` / `genDoMen` | 現場メモ / 原動面 |
| `workerPricingPolicy` | "possible"/"impossible"/"case_by_case" |
| `dailyRateDorm1/2/Commuter` | 寮区分ごとの現場別日給オーバーライド |
| `requiredInsurance` | "any"/"company_only"/"national_only" |
| `workCategory` | "chikuro"=築炉 / "regular" / "spot" |
| `requiredHeadcount` | 案件で必要な人数（注文集計用） |
| `status` | "active"/"completed"/"cancelled" |

**削除**: 物理削除せず `status="cancelled"` にする（過去履歴保護）。

### 6.6 Assignment（配置）

```
id                Int PK
staffId           Int?     null=未割当（現場枠だけ確保、後で人を当てる）
jobSiteId         Int      FK
vehicleId         Int?
startDate / endDate  String  YYYY-MM-DD
assignmentType    "commute"/"business_trip"
shiftType         "day"/"night"
startTime / endTime  String  "08:00"-"18:00"
dailyRateOverride Int?
belongings / contactName / contactTel / transportation / notes
```

**Assignment 1 件は期間を持つ**。実際の日付ごとの状態は **`AssignmentDay`** が個別に保持する（議事録「痛点 #2 解決」）。

#### 6.6.1 AssignmentAllowance（配置単位の加算手当）

路内・出張・食事・とび 等を **自由名称＋金額** で複数件持てる。

```
name      String   "路内手当" / "出張手当" / 自由入力
amount    Int      1日あたり加算
category  "special"=特殊枠 / "other"=他枠   ← 日計表の集計先
```

`category="special"` は日計表の `site*Skill` に、`"other"` は `site*Other` に流れる。

#### 6.6.2 AssignmentDay（配置日）

`(assignmentId, date)` で unique。**配置の核心モデル**。

```
date              String   YYYY-MM-DD
status            "scheduled"/"cancelled"/"completed"/"pre_declined"
startTime / endTime  String?  この日だけの上書き
overtimeHours     Float    残業
dailyRateOverride Int?     この日だけの単価上書き
orderHeadcount    Int?     この日に現場から発注された人数（日毎変動）
notes             String?
acknowledgedAt    DateTime?  スタッフが「見ました」を押した時刻
acknowledgedBy    Int?       押したユーザー ID
```

**Cascade**: `assignment` → `assignmentDays` は ON DELETE CASCADE（Assignment 削除で自動消去）。

### 6.7 Vehicle / WorkCompletionForm / User / AuditLog

- `Vehicle` — `plateNumber` Unique、`inspectionDate`（車検）、`isActive`。Assignment 削除時の cascade は無く、Vehicle 削除時は **配置から vehicleId だけ null に外す**（Assignment は残す）。

- `WorkCompletionForm` — 出来高確認書。`assignmentDayId` 任意（後付け可）、`jobSiteId` 必須、`staffNames` は JSON 文字列、`clientSignature` は base64、`isSubmitted` で編集ロック。

- `User` — `username` Unique、`passwordHash` bcrypt、`role`、`branchOfficeId?`、`staffId?` Unique（1 スタッフ = 1 アカウント上限）、`loginToken` Unique（QR）、`loginTokenAt`、`isActive`。

- `AuditLog` — `userId? username? action(create/update/delete) model recordId diff(JSON 4KB 上限) createdAt`。username は非正規化（User 削除後も誰の操作か残す）。

### 6.8 DailyPayment（日計表）

スタッフ×日付（`@@unique([staffId, date])`）で 1 行。最大 2 現場（site1/site2）を持つ。**派生値（当日残・累計残）は保存しない方針**。

```
date / staffId

## 8. `src/lib/` ユーティリティ層

### 8.1 prisma.ts — DB クライアント＋監査 middleware

3 つのことをやっている:

1. **dual mode 接続**（4.1 参照）
2. **Prisma の `$extends({ query: { $allModels: { $allOperations } } })`** で全ミューテーションをフック
3. ヒットしたら `AsyncLocalStorage` から `actor` を取り出し、`AuditLog` に非同期書き込み（**fire-and-forget**、失敗してもメイン処理を止めない）

監査対象モデル: `Assignment / AssignmentDay / Staff / JobSite / User / WorkCompletionForm / Vehicle`（他は非対象）。

**重要な制約**（コメント `prisma.ts:84-88`）:
> libSQL HTTP adapter ではクエリが直列実行されるため、トランザクション中に `basePrisma.user.findUnique()` を呼ぶとデッドロックする。そのため `setAuditActor()` で AsyncLocalStorage に積まれた actor のみを使い、未設定なら audit log の userId/username は null で記録する。

### 8.2 auth.ts / session.ts

- `session.ts` = iron-session の **設定**
- `auth.ts` = login / loginByToken / logout / getSession の **実装**
- 全ログイン経路で `setAuditActor()` を呼び、以降のリクエスト内 Prisma 操作に actor を伝播

### 8.3 audit-context.ts

`AsyncLocalStorage<AuditActor>` を 1 つだけ持ち、`enterWith()` で「呼び出し以降の同 async コンテキストすべてに伝播」させる方式。`.run()` で wrap しないので route handler の冒頭で呼ぶだけで OK。

```

### 8.6 assignment-validation.ts（配置の検証層）

2 関数:

#### `checkAssignmentConflicts(args)`

並列で 3 つを取得:
1. 自分の指定スタッフが同日に他現場で `scheduled` の `AssignmentDay`
2. スタッフの保険フラグ
3. 現場の `requiredInsurance`

返り値:
- `conflicts`: `{ siteName, dates[] }[]` — 二重ブッキング
- `insuranceWarning`: `{ staffInsurance, siteRequirement, siteName } | null` — 社保のみ／国保のみ要件と不一致
- `vehicleConflicts`: `{ plateNumber, vehicleName, conflictingSiteName, dates[] }[]` — 同日同車両が**別**現場で使われていたら警告（**同一現場での車両共有は OK**：複数人移動の想定）

#### `checkOrderHeadcountOverflow(args)`

「今この配置を保存したら、その日の `scheduled` 数が `orderHeadcount` を超えるか」を検査。返り値 `{ date, orderHeadcount, projectedCount, overflow }[]`。`bulk` でも単発でも使う。

両者とも **`force=true` でユーザーがバイパス可能**（警告は表示するが保存は通す設計）。

### 8.7 payment-utils.ts（日計表ロジック）

`schema.prisma` の `DailyPayment` に対応する純粋計算層。**派生値は DB に保存せず都度計算** がポリシー（`payment-utils.ts:32-34`）。

主要関数:
- `calcPaymentTotal(dp)` = 14 列の支払合計
- `calcOffsetTotal(dp)` = 4 列の相殺合計
- `calcTodayBalance(dp)` = 支払 − 相殺
- `computeCumulativeBalances(staffIds, upToDate, { exclusiveOfDate })` — `openingBalance + Σ todayBalance`（openingBalanceDate 以降のみ）
- `seedDailyPaymentsForDate(date)` — **その日の `AssignmentDay(status="scheduled")` を集約し、`DailyPayment` 行を自動 INSERT / 更新 / 不要な行を削除**

#### `seedDailyPaymentsForDate` の集約ルール（重要）

| 項目 | 優先順位 / 計算 |
|---|---|
| `baseFee` | `assignmentDay.dailyRateOverride` > `assignment.dailyRateOverride` > 現場の寮区分日給 (`dailyRateDorm1/2/Commuter`) > `staff.dailyRate` |
| `skill bonus` | スタッフ保有資格 ∩ 現場 `qualificationBonuses` の合計 + 配置単位の `category="special"` 手当合計 |
| `other bonus` | 配置単位の `category="other"` 手当合計 |
| `lodgingOffset`（**新規行のみ**） | `RESIDENCE_DAILY_COST[residenceType]` (dorm1=1950 / dorm2=1350 / commuter=0) |
| `safetyOffset`（新規行のみ） | **500 円固定で自動計上** |

- スタッフ 1 人につき **最大 2 現場**（site1, site2）まで。3 件以上は `notes` に「3 件以上の配置あり（+N 件は未反映）」と書く。
- **既存行**には `site*Skill` / `site*Other` だけ上書きし、基本料金や相殺は手入力を保持。
- **配置が無くなったスタッフの行**は、`advanceOffset==0 && otherOffset==0 && (notes==null || notes=="")` のときだけ自動削除（手入力痕跡があれば保持）。

#### 累計残の組み立て (`api/daily-payments/[date]/route.ts`)

```
1. seedDailyPaymentsForDate(date)
2. priorBalances = computeCumulativeBalances(staffIds, date, { exclusiveOfDate: true })
       ← 前日末までの累計
3. todayBalance = calcTodayBalance(dp)
4. cumulativeBalance = priorBalance + todayBalance
```

UI 側 (`tally-editor.tsx`) では `priorBalance` を固定値として、編集中の `todayBalance` を都度足し直して即座に再計算。**保存しなくても累計残は画面上で動く**。

なお 2026-05-19 時点で **累計残カラムは「枠だけ残して数値非表示」状態**（編集 / 印刷両画面）。データ計算層 (`computeCumulativeBalances`) は生きており、API レスポンスにも `cumulativeBalance` は含まれる。

## 11. コア・ビジネスロジック

### 11.1 配置の検証（保存時）

```
[POST /api/assignments] or [PUT]
  ↓
checkAssignmentConflicts({ staffId, jobSiteId, dates, vehicleId })
  - 並列で:
      ・staff の他現場 scheduled 配置
      ・staff の保険フラグ
      ・jobSite の requiredInsurance
      ・vehicleId 指定時は他現場で同日同車両
  - 返す: { conflicts[], insuranceWarning?, vehicleConflicts? }
  ↓
checkOrderHeadcountOverflow({ jobSiteId, dates, addedStaffCount, newOrderHeadcount })
  - その日の scheduled 数を集計
  - orderHeadcount を超過する日があれば warning
  ↓
警告ありかつ force=false → 409 + warnings
警告なし or force=true   → 通常作成
```

### 11.2 配置日（AssignmentDay）— 「痛点 #2」の答え

旧来は「期間モデルだけで配置を表現」していたが、
> 「11/15 だけ休み」「11/14 だけ早出」「11/16 だけ事前断り」
が表現できなかった。

そこで Assignment の下に `AssignmentDay` を**日付ごとに 1 行**で持つ。

- **作成時に期間を日数分展開**（POST /api/assignments で `startDate..endDate` を for ループで insert）
- **日別 status**: `scheduled / cancelled / completed / pre_declined`
- **日別 override**: 時刻・単価・人数・備考
- **`acknowledgedAt/By`**: スタッフが「見ました」を押した記録

### 11.3 日計表 — 自動 seed と累計残

§8.7 で詳述したが、ユーザー視点で要約:

1. カレンダーで配置を作る
2. `/tally` を開いたタイミングで `seedDailyPaymentsForDate(date)` が走る
3. その日の `AssignmentDay(scheduled)` を集約して `DailyPayment` 行を**自動生成**
   - `baseFee` = スタッフ日当（オーバーライド優先順位あり）
   - `safetyOffset` = **500 円自動計上**
   - `lodgingOffset` = 寮区分から自動（旧寮 1950 / 新寮 1350 / 通い 0）
4. 既存行は加算手当（skill/other）だけ最新化、基本料金等は手入力を尊重
5. 当日残 = 支払合計 − 相殺合計（**DB に保存しない、都度計算**）
6. 累計残 = `openingBalance + Σ todayBalance`（openingBalanceDate 以降）

**結果**: 配置があれば日計表に金額が自動で乗る。「何も入力していないのに数値が出る」現象の正体はこれ。

### 11.4 出来高確認書 — 印刷ロック

`isSubmitted=true` で UI 編集不可、`POST` ボタンも非活性。施主サインの改ざん防止。

### 11.5 監査ログ

- `auth.ts` 3 経路で `setAuditActor(actor)` → リクエスト内 AsyncLocalStorage に積む
- `prisma.ts` middleware が `$allOperations` で create/update/delete を検知 → 別 promise で `AuditLog.create()`
- 監査対象は 7 モデル（schema.prisma に列挙）。Vehicle/JobSite/Staff/Assignment/AssignmentDay/User/WorkCompletionForm
- `diff` は `{ args, result }` を JSON 化、4KB で打ち切り

### 11.6 QR ログイン

- 管理者が `/users/[id]/qr` で発行 → `loginToken` (base64url 24 byte) を DB に保存
- 印刷物 QR → スマホでスキャン → `/login/qr?token=XXX` → `POST /api/auth/qr` → `loginByToken` で認証 → ロール別ホームへリダイレクト
- 再発行で旧 token は即無効

## 12. UI／UX の設計判断

| 判断 | 理由 |
|---|---|
| **得意先（親）→ 現場（子）の階層を必ず保持**（日計表/印刷/CSV) | 議事録 L131。マスタ化前でも UI 上は崩さない |
| **派生値は DB に保存しない**（当日残・累計残・支払合計） | 過去日編集で全行 ripple update を回避 |
| **削除はマスタ系は基本「論理削除 or 失敗」**（スタッフ→ isActive=false、現場→ status=cancelled、営業所→ 依存あれば 409） | 過去配置のリンク切れ防止 |
| **車両だけは物理削除＋配置から外す** | 「車両自体は捨てたいが配置履歴は残したい」のため |
| **office ロールは金額編集 NG、それ以外は OK** | 5/20 デモの「ユーザー 1（お金関連以外）」概念 |
| **staff ロールは自分の配置と「見ました」のみ** | 配置内容を本人が確認できる以上の権限は与えない |
| **配置 1 件 = 最大 2 現場まで日計表反映、3 件目以降は notes に警告** | 紙の日計表が 2 現場までの様式だったため |
| **safetyOffset 500 デフォルト** | 配置がある日は安全会費 500 円が業務上必ず引かれるため自動計上 |
| **lodgingOffset は新規行のみ自動・既存行は触らない** | 手で変更したかどうか区別できないため、初回だけ自動 |
| **2 文字 + 2 段表記**（印刷の現場/元請） | 紙レイアウトの幅制約 |
| **日計表の累計残カラムは枠だけ残して数値非表示**（2026-05-19） | 自動 seed の影響で「入力してないのに数値」現象が混乱を生むため一時非表示 |
| **モバイル MoreMenu の active 判定は hydrated 後にだけ評価** | Next.js 16 PPR の usePathname 不一致による hydration mismatch 回避 |

## 13. 既知の挙動・注意点・落とし穴

### DB / マイグレーション
- **本番 Turso のスキーマ変更は `turso db shell matsken-prod` で手動 ALTER**（`prisma migrate` は dev.db のみ更新）
- **libSQL HTTP adapter ではトランザクション中の追加 query がデッドロック**するため、`audit-context` 経由で actor を伝播している（DB 引き直しは禁止）
- DailyPayment は `site1Id/site2Id` の FK が `NoAction`。**現場を消す前に DailyPayment.site*Id を null にしないと FK 違反**

### 配置
- `Assignment.staffId` は **nullable**（未割当の現場枠を確保できる）
- POST `/api/assignments` は期間内 **日曜も含めて** AssignmentDay を作る（カレンダー出力では曜日色で識別、CSV では日曜除外）
- `force=true` で警告バイパス可能。UI からは「警告を承知で保存」ボタンで送る

### 日計表
- **何も入力していなくても**、配置と staff.dailyRate と 500 円安全費から自動で数値が乗る
- 累計残は現在非表示。再表示したい時は `tally-editor.tsx` と `tally-print.tsx` の `aria-hidden` 空 `<td>` を元の表示 `<td>` に戻すだけで復活する
- 寮区分日給は **新規 seed 時の `lodgingOffset` 自動セット**にのみ使われる。既存行は触らない
- `site1Lift` は議事録要件で UI 削除予定（互換のため DB 列は残置）

### 印刷
- `/print/assignment/[id]` は **画面で iframe / 印刷で QR コード**にスイッチ。プリント時の `print:` Tailwind variant 利用
- 日計表印刷は A3/A4 切替、営業所別ページ分割。**和暦タイトル**（令和）

### モバイル UI
- `MobileNav` で `useEffect` 後にだけ active 判定する `hydrated` フラグ。**bottom tab 側は未対応**だが現状エラー報告なし

### CSV
- **UTF-8 BOM 必須**（Excel 文字化け対策。議事録で長時間議論された）
- 列順は `csvExportSchema.columns` で UI 指定。デフォルトは 11 項目全部
- **日曜は除外** & 未割当（staffId=null）は除外

### 認証
- iron-session の SESSION_SECRET は **本番では .env から必ず注入**（dev デフォルトは「最低 32 文字」の placeholder）
- QR token は **base64url 24 byte (約 32 文字)**、再発行で旧 token 失効
- staff ロールのスタッフが `User.staffId` を持っていない場合の挙動は実質「個人ページなし」で意味のない状態
